I specialize in Java-based QA automation for web UI and API layers. My goal is to make automated tests useful for real delivery: easy to run, readable during review, stable in CI, and diagnostic when something breaks.

I treat automation as engineering, not only as scripting. A good suite should answer practical questions quickly:

- What changed?
- Which behavior is covered?
- Can a developer reproduce the failure?
- Is the test failure a product issue, an environment issue, or a test design issue?

## How I Work

- I prefer simple framework architecture with clear layers: tests, steps, page objects or API clients, fixtures, configuration, and reporting.
- I use stable selectors, explicit assertions, and controlled waits instead of fragile sleeps.
- I keep smoke tests small and fast, and move broader coverage into regression suites.
- I document tradeoffs so another engineer can maintain the suite after me.
